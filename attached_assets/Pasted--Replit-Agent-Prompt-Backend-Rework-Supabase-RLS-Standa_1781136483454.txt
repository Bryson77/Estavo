## Context

The current `artifacts/api-server` backend uses Drizzle ORM (`lib/db`) with a direct Postgres connection (`DATABASE_URL`) and a custom schema (`users`, `guest_codes`, `gate_logs`, etc.) with custom auth fields (`passwordHash`, `otpCode`). This schema does NOT exist in any live database ŌĆö no migrations have been run.

We are NOT using this Drizzle schema. Instead, we already have a fully set-up Supabase project (`estatehq-app`) with tables, RLS policies, and Storage already configured and live with test data. Environment variables for Supabase (anon key, service role key, platform keys) are already added to this Repl's secrets.

## What to do

### 1. Remove Drizzle entirely
- Delete the `lib/db` package completely (schema files, `drizzle.config.ts`, `index.ts`, `package.json`)
- Remove all imports of `@workspace/db` from `artifacts/api-server`
- Remove `DATABASE_URL` dependency ŌĆö it's not needed

### 2. Use Supabase clients only
`artifacts/api-server/src/lib/supabase.ts` already exports:
- `supabaseApp` ŌĆö service role client (bypasses RLS ŌĆö use ONLY for genuinely cross-estate/admin operations like Superadmin estate provisioning)
- `supabaseAppAnon` ŌĆö anon client, used to verify user JWTs and make RLS-respecting queries on behalf of the logged-in user
- `supabasePlatform` ŌĆö platform project client (billing/superadmin)

For all per-user operations (a resident fetching their own guest codes, a manager viewing their estate's data, etc.), use `supabaseAppAnon` with the user's JWT attached (via `.auth.setSession()` or by passing the JWT in headers, whichever pattern Supabase JS v2 recommends for server-side per-request clients) ŌĆö so RLS policies enforce estate/role isolation automatically.

Use `supabaseApp` (service role) ONLY for: Superadmin "onboard new estate" flow, or other genuinely platform-wide operations explicitly described as such.

- The `requireAuth` middleware should verify the Supabase session JWT (e.g., via `supabaseAppAnon.auth.getUser(token)`) and attach `userId`, `estateId`, `role` (fetched from `profiles`) to the request
